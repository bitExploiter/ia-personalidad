01 / Filosofía
Por qué existe este sistema
La filosofía del Olimpo

El problema central del desarrollo asistido por IA no es la capacidad del modelo — es el contexto. Cuando un agente único carga todo el stack (Go + React + DB + Auth + Tests + CI/CD + Design), el contexto se contamina, las instrucciones se diluyen y la precisión cae.

La solución es especialización con orquestación inteligente. Cada dios del Olimpo carga solo las convenciones de su dominio. Zeus, con el razonamiento superior de Opus, entiende la petición completa, la descompone en responsabilidades atómicas y asigna cada una al dios correcto.

01

Un dios, un dominio

Ningún agente se sale de su jurisdicción. Si una tarea toca dos dominios, Zeus la divide — no la fusiona. Contexto mínimo = máxima precisión.

02

Zeus no ejecuta, dirige

El orquestador razona, planifica y sintetiza. Nunca escribe código ni toca archivos directamente. Ese es el trabajo de los dioses.

03

Paralelismo cuando aplica

Si Apollo puede construir el componente mientras Hermes construye el endpoint, lo hacen al mismo tiempo. Zeus coordina las dependencias.

04

OpenSpec como lingua franca

El ciclo explore→propose→apply→verify→commit→archive es la única forma válida de trabajar. Los artefactos son el bus de contexto compartido entre todos los dioses.

05

Calidad en tres niveles

Artemis garantiza la calidad por capa (unit + integration). Hera garantiza la integración entre capas (E2E, contratos). Hestia garantiza que todo quede documentado y auditado.

06

Verificación siempre

Zeus nunca cierra una tarea sin confirmación externa: visual en Chrome, curl al endpoint, tests verdes. La regla es para todos los dioses sin excepción.

05 / OpenSpec como Bus de Contexto
Contexto compartido entre agentes
OpenSpec garantiza la coherencia

El desafío de un sistema multiagente es que cada dios trabaja con contexto especializado. Sin un mecanismo de sincronización, Hermes puede construir un endpoint con una forma distinta a la que Apollo espera. Los artefactos de OpenSpec son el bus de contexto compartido que elimina esa ambigüedad.

⚡ La Regla de Oro

Ningún dios empieza a ejecutar sin un OpenSpec change activo con specs.md aprobado por Zeus. specs.md es el contrato. Hermes lo implementa por el lado del servidor. Apollo por el lado del cliente. Hera lo usa para validar que ambos lados lo honraron. Hestia lo convierte en documentación permanente.

Los 4 Artefactos del Change
📋 proposal.md
Qué se construye y por qué. La motivación, el alcance y los criterios de éxito del change.
Zeus Hestia (documenta el "por qué")
📐 specs.md
El contrato técnico completo: endpoints exactos, schemas de DB, tipos compartidos, códigos de error y su forma JSON.
Hermes Apollo Demeter Athena Artemis Hera Hestia
🎨 design.md
Prompt de Stitch informado por specs + wireframe. Solo cuando hay UI nueva. Schema → API → Prompt → Diseño.
Hephaestus Apollo
✅ tasks.md
Lista de tareas asignadas por dios con definición de hecho medible. Un checkbox por tarea concreta y verificable.
Zeus (asigna) Artemis (valida) Hera (valida integración) Hestia (audita cumplimiento)
1

/opsx:explore — Zeus asigna al dios experto del área

Antes de proponer nada, Zeus convoca al dios que conoce el área afectada para que explore el código existente y reporte qué hay, qué falta y qué puede colisionar.

2

/opsx:propose — Zeus crea el change con los 4 artefactos

Con el resultado del explore, Zeus redacta proposal.md, specs.md, design.md (si aplica) y tasks.md. Todos los dioses que van a trabajar reciben specs.md en su briefing.

3

/opsx:apply — Cada dios ejecuta su sección de tasks.md

Zeus briefea a cada dios: su tarea específica + specs.md completo + sus archivos IA/. Los dioses sin dependencias entre sí ejecutan en paralelo.

4

Artemis valida + Hera integra + Hestia audita

Artemis verifica contra specs.md que cada capa implementó lo que prometía. Hera valida la integración de punta a punta. Hestia audita documentación y seguridad.

5

Commit + /opsx:archive — Zeus cierra el ciclo

Un commit semántico por capa lógica. Zeus archiva el change. Hera ha emitido la señal DONE. Zeus reporta al usuario el resumen completo: qué se hizo, qué se verificó, qué quedó pendiente.

Zeus recibe: "Quiero un módulo de gestión de usuarios con roles admin y user"

Datos

FASE 2
Backend
(paralelo)
🦉 Athena
🪶 Hermes

FASE 3
Frontend
(paralelo)
☀️ Apollo

FASE 4
QA Capa
(paralelo)
🏹 Artemis Frontend
🏹 Artemis Backend

FASE 5
Integración
👑 Hera
Valida contratos API, flujos E2E completos (login → crear usuario → asignar rol → verificar permisos). ¿El Zod schema del frontend coincide con lo que manda Hermes? ¿Los errores de Athena se manejan en el frontend?
→ Emite señal DONE a Zeus ✓

Infra
🔱 Poseidon
Verificar variables de entorno en CI/CD. Actualizar docker-compose si hay nuevo servicio. Confirmar que el pipeline incluye los nuevos tests. Commit + PR a testing.
CIERRE
⚡ Zeus
Commits semánticos por capa. /opsx:archive. Reporte final al usuario: qué se hizo, qué se verificó, qué quedó pendiente.

Qué no hacer
Anti-patrones del sistema
❌ El usuario habla directo con Apollo o Hermes
Zeus es el único punto de entrada. Siempre. Los dioses no tienen interfaz pública.
❌ Zeus escribe código
Zeus razona y coordina. El código lo escriben los dioses. Zeus que implementa es Zeus que pierde capacidad de orquestación.
❌ Un dios carga todos los archivos IA/
El contexto especializado es el punto. Cada dios carga solo lo suyo. Si Apollo carga go.md, contamina su contexto.
❌ Hera valida sin que Artemis haya pasado
El orden importa: Artemis valida por capas primero. Hera valida la integración. No se puede saltar la QA por capa.
❌ Ejecutar sin specs.md aprobado
Sin el contrato técnico, Hermes y Apollo implementan algo diferente. specs.md es obligatorio antes de cualquier apply.
❌ Hestia bloquea el flujo
Hestia es transversal — corre en paralelo, no en secuencia. Nunca debe ser un cuello de botella entre fases.
